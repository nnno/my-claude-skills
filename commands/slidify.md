# スライド化セッション (HTMLモード)

**ゼロベースから対話で** HTML 登壇資料を作る。素材ファイルや knowledge を入力に取らず、ユーザーとの問答で内容そのものを引き出しながら進める。

引数: $ARGUMENTS （任意。「テーマ」だけ与えてもよい。空でもよい）

## 出力形式の選択

| モード | 場所 | 用途 |
|--------|------|------|
| **HTML 登壇資料 (このコマンド)** | nnno-bookmark | 個人発表・社内LT・登壇・ブログ素材。単一HTMLで完結、ブラウザでプレゼン |
| **PPTX 提案書** | `/Users/nnno/work/make-presentation` の `/slidify` | 顧客提案書・編集を渡す資料・印刷配布 |

両者は同じ 5 原則 (3フェーズ分離 / 1スライド1メッセージ / 中間アーティファクト / オーバーフロー検出 / テーマ資産再利用) に従う。出力経路だけが異なる。

## このスキルの立ち位置

このスキルは **オーケストレーター**。中身を書くスキルではない。下記の責務だけを担う:

1. ユーザーから内容を **対話で引き出す**
2. 各フェーズの成果を **Markdown ファイル** として残す
3. 各フェーズの **ゲート** で停止し、ユーザー承認を取る
4. レンダリングは `frontend-slides` skill に委譲

「素材ファイルがあるならまず読ませて」とユーザーが言った場合のみ、Phase 1 の補助として参考にする。スタート地点は常に **白紙** と前提する。

## 5原則（拝借元: Spec-Driven Presentation Maker）

1. **3フェーズ強制分離**: briefing → outline → art direction を順に。スキップ禁止
2. **1スライド1メッセージ**: outline段階で明文化。2つ乗りそうなら分割
3. **中間アーティファクトを残す**: 各フェーズの成果を Markdown ファイルとして保存
4. **オーバーフロー検出**: viewport-fit を CSS で強制、複数解像度でチェック
5. **CSS資産の再利用**: `presentations/_assets/` を優先、新規分は資産化

## ディレクトリ構成

```
presentations/
├── YYYY-MM-DD_<slug>/
│   ├── briefing.md
│   ├── outline.md
│   ├── art-direction.md
│   ├── previews/         # Phase 3 のスタイル探索（一時）
│   └── slides.html
└── _assets/
    ├── tokens.css
    ├── theme-<name>.css
    └── fonts/
```

## 手順

### Phase 0: 始動

1. **挨拶 + 引数の解釈**
   - `$ARGUMENTS` が空 → 「今日は何のスライドを作りますか？」と問う
   - `$ARGUMENTS` にキーワード（例: 「RAG入門」「社内LT」）がある → 「`<X>` の話を一緒に組み立てましょう。まず聞かせてください」と受ける
   - **素材ファイルを指定された場合のみ**「参考に読み込んでも？」と確認してから読む。それ以外は読み込まない
2. **目的の輪郭を 1 ターンで取る**
   - 「これは登壇？提案？社内共有？教育？」「持ち時間は？」「1枚で何を伝える資料？」
   - 1ターン最大3問。深追いしない。これは Phase 1 への助走

> ⚠️ Phase 0 でディレクトリは **まだ作らない**。タイトルが decideable になってから。

### Phase 1: ブリーフィング（briefing.md）

**目的**: 「誰に・何を伝え・何を起こすか」を強制言語化。**この段階で内容そのものを掘る**。

#### 対話の進め方

ユーザーに対し、以下を **順番に・1 ターン 1〜2 質問** で引き出す。
書いてあるものを並べるのではなく、未言語化のものを引き出す。

1. **オーディエンス**
   - 「聞き手は誰？職種・人数・前提知識のレベル」
   - 「その人たちが今、似たトピックでよく見るものは？」（前提知識の解像度を上げる質問）
2. **シーン**
   - 「いつ・どこで話す？プロジェクター？Zoom？ブログ埋め込み？」
   - 「持ち時間と、その中で質疑にどれくらい取りたい？」
3. **ゴール**
   - 「聞き終わって聞き手に何を持って帰ってほしい？」
   - 「行動を促すなら何の行動？」
4. **コア主張**
   - 「もし 1 行だけ覚えてもらえるなら、何？」
   - これが空欄のまま次に行かない。**ゲート**
5. **トーン**
   - 「真面目寄り？砕け寄り？皮肉あり？数字推し？」
   - 「自分の声として一番近い既存資料は？」（あれば参考にする）

#### briefing.md テンプレ

```markdown
# Briefing — <タイトル>

## オーディエンス
- 職種・人数・前提知識
- 関心事

## シーン
- 形態（登壇 / 提案 / 社内 / ブログ）
- 場所・時間・質疑

## ゴール
- 持ち帰ってほしいこと（1〜2 行）
- 起こしたい行動

## コア主張（1 行）
**<X すると Y になる>**

## トーン
- フォーマル度
- 声
- 禁則表現
```

#### ゲート
- コア主張が「主題のみ」ではなく「結論」になっていること
- ゴールが「次の行動」に翻訳できていること

OKならディレクトリ `presentations/YYYY-MM-DD_<slug>/` を作って `briefing.md` を保存。

### Phase 2: アウトライン（outline.md）

**目的**: 1スライド1メッセージで構成設計。**ビジュアルはまだ考えない**。

#### 対話の進め方

1. **スキル側から最初のドラフトを出す**
   - briefing.md を元に「結論→根拠→対比→行動」のような全体アークを 5〜10 枚で提案する
   - 提案は表で出す（# / メッセージ / 補助要素）
2. **ユーザーに修正点を聞く**
   - 「足りない章、削る章、入れ替えたい順番は？」
   - 1往復で fix する想定。深入りしない
3. **コードや図解の必要性を判定**
   - 各スライドに「補助要素: なし / 図解 / コード / 表 / 引用」をタグ付け
4. **多メッセージ違反をチェック**
   - 1スライドに「Aであり、かつBである」のような並立があれば分割を提案
5. **保存**: `outline.md`

#### outline.md テンプレ

```markdown
# Outline — <タイトル>

## 全体アーク
<5-10 枚の流れを 2-3 文で>

## スライド一覧

| # | メッセージ（1つだけ） | 根拠 / データ | 補助要素 |
|---|--------------------|-------------|--------|
| 1 | ... | ... | なし |
| 2 | ... | ... | 図解 |

## 構造的工夫
- 伏線 / 回収 / 対比 / 3点法 のどれを使うか

## 1メッセージ違反チェック
- [ ] 各行のメッセージ列に並立構造がない
```

#### ゲート
- スライド数が briefing の想定時間に対して妥当か（1 枚 ≒ 1〜2 分が目安）
- コア主張が「最初」または「最後」に明示的に置かれているか

### Phase 3: アートディレクション（art-direction.md）

**目的**: 配色・タイポ・雰囲気を決める。**スライド本体はまだ作らない**。

#### 対話の進め方（視覚探索を優先）

「どんな雰囲気にしたい？」と抽象質問しても答えが返らないので、**3 枚のプレビューを並べて選ばせる**。

1. **既存資産を確認**
   - `presentations/_assets/` を ls。既存テーマがあれば候補に
2. **雰囲気の方向性を聞く（1 ターン）**
   - 「impressed / energized / focused / inspired のどれが近い？」
   - frontend-slides skill の `STYLE_PRESETS.md` を参照、当てはまるプリセットを 2-3 個ピック
3. **3 枚のプレビュー生成**
   - `presentations/YYYY-MM-DD_<slug>/previews/` に `preview-<name>.html` を 3 つ作る
   - 各プレビューは **outline.md の最初のスライド 1 枚分**を、異なるテーマで描いたもの
   - 自分の判断で勝手に決めない。必ず 3 案出してユーザーに選ばせる
4. **ユーザーが選んだら art-direction.md に確定**
   - 「採用プリセット名」「派生があれば差分」「フォント」「動きの方針」を記録
5. **previews/ は最後に削除** （資産化したいテーマがあれば `_assets/` に切り出してから）

#### art-direction.md テンプレ

```markdown
# Art Direction — <タイトル>

## 採用プリセット
**<プリセット名>**（frontend-slides STYLE_PRESETS より）

## 選定理由
briefing の「<トーン>」と「<コア主張>」に基づく:
→ <1-2 文で>

## カラートークン
- surface / text / accent / mute

## タイポグラフィ
- 見出し: <font>
- 本文: <font>

## モーション方針
- フェード / スライド / 静止

## 情報密度
- bullets 上限: <既定 6>
- 1枚あたり情報量の目安

## 既存資産との関係
- `_assets/theme-X.css` を流用 / 派生 / 新規
```

#### ゲート
- ユーザーがプレビューを **実際に見た**上で決めたか（聞いただけで決めない）

### Phase 4: レンダリング（slides.html）

**目的**: `frontend-slides` skill にレンダリングを委譲。

1. `frontend-slides` skill を呼ぶ
2. 入力として渡すもの:
   - `outline.md`（構成）
   - `art-direction.md`（テーマ宣言）
   - `_assets/` の流用CSSがあればパス
3. 単一HTMLとして `presentations/YYYY-MM-DD_<slug>/slides.html` に出力
4. 画像等の追加 assets はそのディレクトリ配下に

### Phase 5: 検証

**目的**: SDPM の `measure_slides` 相当を担保。

#### 静的チェック（必ず実施）

`slides.html` を grep で機械的に検証:

| 観点 | チェック内容 |
|------|------------|
| viewport-fit | `height: 100vh` / `100dvh` / `overflow: hidden` の存在 |
| 流体スケール | `clamp(` の利用箇所 |
| 動きの配慮 | `prefers-reduced-motion` の対応 |
| ナビ | keydown / touchstart / wheel ハンドラの存在 |
| 地雷構文 | `-clamp(` `-min(` のような負値構文が **無い** こと |

#### ブラウザ確認（可能なら）

Playwright が使えるなら以下を実行:
- 解像度 1920x1080 / 1280x720 / 768x1024 で各スライドのスクリーンショット
- スクロール量を測り、いずれかのスライドが viewport を超えていないか確認

サンドボックスや環境でブラウザが使えない場合、その旨をユーザーに告げ、`! open <path>` での手動確認を案内する。

#### はみ出しを検出したら

- `outline.md` に戻り、対象スライドを **分割する** か **文字量を削る**
- **フォントを小さくして逃げない**

### Phase 6: 資産化と引き渡し

1. **previews/ を削除**（保持指定があれば残す）
2. **テーマ資産化の判定**
   - 新規作成したテーマCSSやコンポーネントを `_assets/` に切り出すか聞く
   - 切り出す場合は `_assets/theme-<name>.css` などに分離
3. **サマリー報告**
   - ファイルパス
   - スライド枚数
   - 使用プリセット
   - 静的チェック結果
   - 「`! open <path>`」で開ける旨

## 注意事項

- **白紙からのドラフトを引き出す**ことに価値がある。素材ファイルを読み込んで要約する作業ではない
- **3フェーズは必ず順番に**。いきなり Phase 3 や Phase 4 にジャンプしない
- **briefing でコア主張が言語化できなかったら止まる**。「未確定」のまま走らない
- **Phase 3 は 3 案を見せる**。1 案を押し付けない（ユーザーがプリセット名を即答できる場合は除く）
- **過度な質問攻めは禁止**。1 ターン 1〜2 問が原則
- **既存の knowledge ファイルを参考にしたい**とユーザーが明示的に言った場合のみ読み込む。デフォルトでは読まない
