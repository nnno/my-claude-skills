# レポートテンプレート

以下のテンプレートに従ってレポートを出力する。

```markdown
# 脆弱性リスク評価レポート

## 概要

| 項目 | 値 |
|---|---|
| CVE ID | CVE-XXXX-XXXXX |
| 対象パッケージ | {パッケージ名} {バージョン} |
| Inspector2重要度 | {CRITICAL/HIGH/MEDIUM/LOW} |
| **実質リスクレベル** | **{Critical/High/Medium/Low/Informational}** |
| 評価日 | YYYY-MM-DD |

## CVE詳細

- **CVSSスコア**: X.X ({ベクトル文字列})
- **攻撃ベクトル**: {Network/Adjacent/Local/Physical}
- **攻撃複雑性**: {Low/High}
- **必要権限**: {None/Low/High}
- **脆弱性の種類**: {RCE/XSS/SQLi/DoS/情報漏洩/権限昇格/その他}
- **概要**: {脆弱性の技術的説明を1-2文で}

## ソースコード分析

### 依存関係
- **使用バージョン**: {バージョン}（{依存定義ファイルのパス}）
- **修正バージョン**: {修正済みバージョン}
- **影響バージョン範囲**: {範囲}

### 使用箇所
{該当するコード箇所を列挙。ファイルパス:行番号の形式で記載}

- `path/to/file.py:42` — {使用内容の説明}
- `path/to/other.py:108` — {使用内容の説明}

使用箇所が見つからない場合: 「脆弱な機能の直接的な使用箇所は確認されなかった」

### 外部入力の到達可能性
{攻撃に必要な入力が脆弱な関数に到達するかの分析結果}

## リスク評価

### Inspector2重要度との差分
{Inspector2の重要度と実質リスクレベルが異なる場合、その理由を説明}

### 判定根拠
{以下の観点での評価結果を箇条書きで記載}
- 脆弱な機能の使用有無:
- 外部入力の到達可能性:
- 攻撃前提条件の充足状況:
- 環境要因（ネットワーク構成等）:

## 推奨対応

### 修正方法
{具体的な修正手順。以下のいずれかまたは組み合わせ}

**パッケージアップデートの場合:**
1. {パッケージ名}を{修正バージョン}以上にアップデート
2. 変更コマンド例: `{npm install / pip install / go get 等}`
3. 破壊的変更の有無: {あり/なし。ありの場合は影響箇所を記載}

**コード修正の場合:**
1. {具体的な修正内容}
2. 修正対象ファイル: {パス}
3. 修正案:
   ```{lang}
   // Before
   {修正前コード}

   // After
   {修正後コード}
   ```

**緩和策の場合（即時修正が困難な場合）:**
1. {WAFルール追加、入力バリデーション強化等の緩和策}

### 対応優先度
- **緊急（即時対応）**: Critical — 攻撃が容易で影響が大きい
- **高（1週間以内）**: High — 攻撃可能だが条件がある
- **中（次回リリース）**: Medium — 限定的な影響
- **低（計画的対応）**: Low/Informational — 実質的影響が小さい
```
