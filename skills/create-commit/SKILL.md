# コミットメッセージ生成

ステージング済みの変更を分析し、Conventional Commits形式のコミットメッセージを生成します。

## モード

このスキルは2つのモードで動作します。

| モード | 起動方法 | 挙動 |
|--------|----------|------|
| interactive（既定） | `/create-commit`（引数なし） | 生成メッセージをユーザーに提示し、承認後にコミット |
| auto | `/create-commit auto` | 機械的ガードレールを通過したら承認なしでコミット |

**モード判定ロジック**:

- 起動時の引数（Skill tool の `args`）が空 / 未指定 → interactive モード
- `args` に `auto` が含まれる → auto モード
- 上記以外の文字列が渡された → 解釈できない旨を伝えて中止

## プロジェクトルールの参照（両モード共通）

コミットメッセージを生成する前に、以下を確認してください：

1. commitlint 設定ファイルがあれば参照する（必須ルール、優先度順）：
   - `commitlint.config.js` / `commitlint.config.cjs` / `commitlint.config.mjs`
   - `.commitlintrc.js` / `.commitlintrc.json` / `.commitlintrc.yml`
   - `package.json` の `commitlint` フィールド
2. プロジェクトの `CLAUDE.md` / `AGENTS.md` にコミットメッセージ規約があるか確認（補足ガイドライン）
3. プロジェクト固有のルールがこのコマンドのデフォルトルールと異なる場合は、プロジェクトのルールを優先する

## 実行手順

### 1. ステージング状態の確認

まずステージング済みの変更があるか確認します：

```bash
git status
```

**ステージング済みの変更がない場合は、処理を中断してユーザーに `git add` を促してください。**（両モード共通）

### 2. 変更内容の分析

ステージング済みの差分を取得します：

```bash
git diff --cached
```

ファイル名のみ取得（ガードレール検査と scope 推定で使用）：

```bash
git diff --cached --name-only
```

### 3. コミットメッセージの生成（両モード共通）

差分を分析し、以下のルールに従ってコミットメッセージを生成してください。

#### フォーマット

```
<type>(<scope>): <subject>
```

#### type（変更の種類）

| type | 用途 |
|------|------|
| `feat` | 新機能の追加 |
| `fix` | バグ修正 |
| `docs` | ドキュメントの変更 |
| `style` | コードフォーマット、セミコロン等（動作に影響しない変更） |
| `refactor` | リファクタリング（機能追加・バグ修正を含まない） |
| `perf` | パフォーマンス改善 |
| `test` | テストの追加・修正 |
| `chore` | ビルドプロセス、補助ツールの変更 |
| `ci` | CI設定の変更 |

#### scope（影響範囲）

- 原則必須
- 変更対象のモジュール名、機能名、ファイル名などを記載
- 複数ファイルにまたがる場合は、最も代表的な範囲を選択
- 例: `auth`, `api`, `ui`, `config`, `db`

#### subject（変更内容の要約）

- 日本語で記述
- 50文字以内を目安
- 体言止めまたは動詞で終わる簡潔な表現
- 例: 「ログイン機能を追加」「バリデーションエラーを修正」

### 4. モードごとの分岐

#### interactive モード（既定）

生成したコミットメッセージを以下の形式で提案してください：

```
## 提案するコミットメッセージ

<生成したコミットメッセージ>

このメッセージでコミットしてよろしいですか？
- 「はい」または「y」で実行
- 修正したい場合は修正内容をお伝えください
```

ユーザーの承認を得た後にのみ、コミットを実行します：

```bash
git commit -m "<承認されたコミットメッセージ>"
```

メッセージの修正依頼があった場合は、修正版を再提案してください。

#### auto モード

auto モードでは、ステップ 4-A → 4-B → 4-C の順で進めます。**いずれかのチェックで違反を検知した場合はコミットせず中止**してください。

##### 4-A. 機密ファイル検知

`git diff --cached --name-only` の出力に対し、以下のパターンに**一つでも**マッチするファイルがあれば中止:

- `.env`、`.env.*`
  - ただし `.env.example` / `.env.sample` / `.env.template` は許可
- `*.pem`、`*.key`、`id_rsa`、`id_ed25519`、`id_dsa`、`id_ecdsa`
- `credentials*`、`secrets*`、`*.kdbx`
- `service-account*.json`、`gcp-key*.json`、`*-credentials.json`

判定はモデル側で行います。誤検知が起きやすい曖昧なケース（例: ドキュメント内の `secrets-management.md`、テストフィクスチャの `dummy.pem`）は文脈から許可可否を判断してよいが、**判断に迷う場合は安全側に倒し中止すること**。

##### 4-B. マージ競合マーカー検知

`git diff --cached` の出力（追加行）に、以下のいずれかの形式の行が含まれていれば中止:

- `<<<<<<< ` で始まる行（7 文字 + 半角スペース）
- `>>>>>>> ` で始まる行（7 文字 + 半角スペース）
- `=======` のみの行（前後に空白以外なし）

diff のコンテキスト行や、文字列リテラル内の偶発一致と区別するため、**追加行（`+` で始まる行）の中身**に対してのみマッチさせること。

##### 4-C. コミット実行

ガードレール通過時のみコミット。**実行前に生成したメッセージとガードレール結果を必ず transcript に出力**してください（事後レビュー可能性のため）:

```
## auto モードでコミット実行

メッセージ:
<生成したコミットメッセージ>

ガードレール: secret OK / conflict OK
```

その後コミットを実行：

```bash
git commit -m "<生成したコミットメッセージ>"
```

##### 違反時の挙動

ガードレール違反を検知した場合は **コミットを実行せず**、以下の構造で出力して終了してください：

```
## auto モード中止

理由: <secret 検知 / conflict マーカー検知>
詳細: <該当ファイル名 or 該当箇所>

対応:
- 該当ファイルをステージングから外す（git reset HEAD <file>）
- または対応後に `/create-commit`（auto なし）で人間承認フローを再実行
```

interactive モードへの自動フォールバックはしません（auto モードに違反があった事実を呼び出し側に明示するため）。

## 注意事項

- ステージングされていない変更は対象外です
- interactive モードでは、ユーザーの明示的な承認なしにコミットを実行しないでください
- auto モードはあくまで「承認ステップの省略」です。**メッセージ生成ルール（type/scope/subject、日本語、Conventional Commits prefix 等）は両モードで不変**です
- プロジェクト固有のメッセージ規約（例: 日本語コミットメッセージ）が CLAUDE.md / AGENTS.md に定義されている場合、auto モードでもそれに従ってください
- auto モードのガードレールは secret + conflict のみです。ブランチ保護やテスト合格などの他の条件は対象外なので、必要に応じて呼び出し側で別途担保してください
